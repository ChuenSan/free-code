# Findings (per-PR)

### PR #7 - Context collapse and snip compaction

#### P1 - `HISTORY_SNIP` callers treat an async compaction result as synchronous

- `verdict`: CONFIRMED
- Anchors: `src/services/compact/snipCompact.ts:339`
- Scenario: with `HISTORY_SNIP` enabled, `snipCompactIfNeeded()` now returns `Promise<SnipResult>`, while the existing snip integration consumes this helper as an immediate `SnipResult`. The unresolved Promise has no `messages` or `tokensFreed` fields, so the message pipeline can receive `undefined` before microcompact/autocompact.
- Impact: snip-enabled turns can crash or corrupt the message pipeline before the API call.

#### P1 - overflow recovery is async but the 413 recovery path never awaits it or passes the required signal

- `verdict`: CONFIRMED
- Anchors: `src/services/contextCollapse/index.ts:163`, `src/services/contextCollapse/index.ts:172`
- Scenario: on a withheld 413, the existing recovery integration expects `recoverFromOverflow(...)` to synchronously return `{ messages, committed }`, but this diff changes it to return `Promise<RecoveryResult>` and also requires `toolUseContext.abortController.signal` to do work. A caller that does not await it sees no `committed` value, and a caller that omits `toolUseContext` always gets `{ committed: 0 }`.
- Impact: context-collapse drain retry never runs, so large contexts fall through to heavier recovery or still fail 413.

### PR #8 - Third-party provider support

#### P1 - cached microcompact now runs first-party cache-editing behavior for third-party providers

- `verdict`: CONFIRMED
- Anchors: `src/services/api/claude.ts:1435`, `src/services/api/claude.ts:1674`, `src/services/api/claude.ts:1682`, `src/services/api/openaiCompatible.ts:123`
- Scenario: with `CLAUDE_CODE_USE_OPENAI=1` and cached microcompact enabled on the REPL thread, `claude.ts` latches the cache-edit beta and passes `useCachedMC=true` without checking `getAPIProvider() === 'firstParty'`. The OpenAI-compatible translator only preserves `text` and `tool_result` user blocks, so `cache_edits` blocks inserted by `addCacheBreakpoints()` are dropped while local cached-microcompact state has already consumed/pinned them.
- Impact: third-party requests can diverge from local cache-edit state, causing cache deletion accounting and prompt construction to be wrong.

### PR #28 - vLLM provider

#### P1 - `/provider` cannot select or clear vLLM, so provider switches can silently leave vLLM active

- `verdict`: CONFIRMED
- Anchors: `src/commands/provider/provider.ts:7`, `src/commands/provider/provider.ts:15`, `src/commands/provider/provider.ts:27`, `src/utils/model/providers.ts:20`
- Scenario: a user starts with `CLAUDE_CODE_USE_VLLM=1`, then runs `/provider openai`. `providerEnv()` sets OpenAI/Copilot/Bedrock/Vertex/Foundry flags but never writes `CLAUDE_CODE_USE_VLLM=0`. `getAPIProvider()` checks `CLAUDE_CODE_USE_VLLM` before OpenAI, so the command reports a switch while requests still route to vLLM.
- Impact: provider selection is misleading and can send prompts to the wrong endpoint.

### PR #34 - package/provider integration

#### P1 - Copilot routing env vars are not host-managed

- `verdict`: CONFIRMED
- Anchors: `src/utils/managedEnvConstants.ts:17`, `src/utils/managedEnvConstants.ts:36`, `src/services/api/client.ts:364`, `src/services/api/client.ts:365`
- Scenario: a host starts the process with `CLAUDE_CODE_PROVIDER_MANAGED_BY_HOST=1` expecting settings-sourced provider config to be stripped. The managed set includes vLLM and OpenAI flags/keys but omits `CLAUDE_CODE_USE_COPILOT`, `COPILOT_TOKEN`, and `COPILOT_BASE_URL`. A user setting can therefore force Copilot routing or replace the Copilot token/base URL despite host-managed routing.
- Impact: host-managed provider isolation is incomplete for Copilot.

### PR #41 - Windows installer and key purge

#### P1 - Windows installer executes a remote installer script without pinning or verification

- `verdict`: CONFIRMED
- Anchors: `install.ps1:86`, `install.ps1:89`
- Scenario: on a Windows machine without Bun, the installer runs `irm https://bun.sh/install.ps1 | iex`, then falls back to `Invoke-RestMethod ... | Invoke-Expression`. If that remote script, CDN path, or TLS interception point is compromised, arbitrary PowerShell runs in the user's shell before the repository build starts.
- Impact: installation has an avoidable supply-chain remote-code-execution risk.

#### P1 - `windows-install` package script points to a non-existent file

- `verdict`: CONFIRMED
- Anchors: `package.json:21`, `install.ps1:1`
- Scenario: `bun run windows-install` executes `pwsh -File scripts/install.ps1`, but the diff adds `install.ps1` at the repository root, not under `scripts/`. On Windows this command fails before performing any install.
- Impact: the advertised package script for Windows install is broken.
